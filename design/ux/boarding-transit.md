# UX Spec: 上车与在途 (Boarding & Transit · 派车单)

> **Status**: Drafted（待 PM 终审 2026-06-11）
> **Author**: Yan Dai + ux-designer
> **Last Updated**: 2026-06-11
> **Journey Phase(s)**: 出发（HQ→任务点）与返程（任务点→HQ）两段在途（无 player-journey.md，按核心循环推定）
> **Template**: UX Spec（流程型：覆盖上车确认 + 车厢即加载 + 提前收工申请三段同链）
> **统一语言定位**: 三类表面之「盖章公文卡」×2（上车=派车单、部分返程=提前收工申请单）+ 现场票据条
> **队列**: UI 重设计第 4 屏（电脑✅ → HUD✅ → 结算✅已实施 → **上车+加载** → 大厅 → 图谱 → 设置）

## Purpose & Player Need

派遣车仪式是本作的支柱体验：上车、等齐人、签发、在途、开门——每次委托的呼吸节拍。
本流程同时承担三件事：**①集合确认**（"人齐了没？谁还在拿手电？"）**②加载伪装**
（PM 锁定 2026-06-11：在途时间=异步场景加载窗口，到站开门，全链无黑屏无 2D 加载页）
**③不可逆决策的防误触**（提前收工拉全队回所，必须是郑重的、有成本的动作）。
没有这一屏，加载就是黑屏硬切，提前收工就是手滑双击。

**PM 四锁定（2026-06-11）**: ①派车单=混合形态（等人=顶部票据条，全员就位才弹中央卡）
②在途进度=里程虚构+最短仪式（与真实加载解耦）③提前收工=房主按住 E 1.2s 签字
④到站=开门渗光+自行下车（光色即地点签名）。

## Player Context on Arrival

**出发段（HQ→任务点）**
- **时点**: 办公电脑接单（终端 spec [1]委托文件）→ 领装备 → 走向车库 → E 上车。
- **情绪假设**: 期待+备战。多人时有"催队友"的集合社交（谁没上车一目了然）。
- **到达方式**: 自愿——上车是玩家主动动作；发车是房主主动签发。

**返程段（任务点→HQ）**
- **时点**: 三种终态入口——拉杆/Space 完整发车、未取柱房主申请提前收工、全员倒地强制回收。
- **情绪假设**: 肾上腺素回落。在途车厢是被捕获时间：读结算单（结算 spec）、复盘聊天。
- **到达方式**: 完整返程自愿；失败强制（无上车环节，直接入舱）。

---

## Navigation Position

不是导航目的地，是核心循环的物理通道，全部入口都是世界内交互：
`HQ 车库车尾 [E] → 车厢（等人→派车单→在途）→ 任务点出生`
`任务点车尾 [E] → 车厢（→派车单/提前收工申请单→在途+结算单）→ HQ 车库`
无任何菜单可主动进入；车厢内唯一"返回"是发车前 [X] 离座。

## Entry & Exit Points

| Entry Source | Trigger | Player carries this context |
|---|---|---|
| HQ 车库（出发） | `OfficeDepartureVan` E 上车 | 已选委托（终端接单）、装备在热栏 |
| 任务点车尾（返程） | `MissionVanExitPoint` E「上车/上车返程」 | 终态前状态：取柱与否、完整度、储物柜补给 |
| 全员倒地（强制） | all-downed → Failed | 无上车环节；车厢+结算单直接出现（事故回收，注：演出从简，见 Open Questions） |

| Exit Destination | Trigger | Notes |
|---|---|---|
| 离座（发车前） | `X` | 仅 Boarding 阶段有效；在途锁定 |
| 任务点出生（出发到站） | 加载完成+最短在途走完 → 开门 → E/走出 | 不可逆点=房主 Space 签发那一刻 |
| HQ 车库（返程到站） | 同上 | 结算单若展开随下车自动收起（结算 spec） |
| 撤回收工申请 | `Esc`（签字完成前） | 卡收起，回 Boarding 等待状态，可反复 |

不可逆点明确化：**出发=签发章落下**（之后 X 无效）；**返程=申请单签字墨条满**或
**完整返程 Space**（拉杆同义）。签字完成前一切可撤回。

## Layout Specification

### Information Hierarchy

1. **就位状态 / 发车许可** — "还差谁？能不能走？"（等人期的唯一焦点）
2. **委托名 + 目的地** — 这趟车去哪、为什么（票据条与派车单共有）
3. **操作指引** — Space 签发 / X 离座 / 按住 E 签字 / Esc 撤回
4. **在途进度 + 预计时间** — 多人等待焦虑的解药
5. **警告信息**（仅收工申请单）— 折算金额、拒收风险、全队随车

### Layout Zones

三个表面，分阶段接力，全部叠在真实 3D 车厢之上：

- **顶部票据条**（贯穿全程）: 屏幕顶部居中，纸底+市政青框线+红章块（现实装样式保留）。
  等人期=就位计数+键提示；在途期=里程进度；到站=「已抵达」收尾。
- **中央派车单**（全员就位→签发后随章收起）: 竖版盖章公文卡，屏幕中央。
- **中央提前收工申请单**（房主 Space 且未取柱时）: 同卡体语法，警示内容。
- 车厢本体（3D 舱体、电台、灯）不属于本 spec 的 UI 层，但到站开门演出依赖它。

### 车厢内饰美术方向（LOCKED 2026-06-13）

> 3D 内饰的"长相"不在本 UI spec 的画面层，但 PM 已锁定美术目标，记录于此供实施与 art-bible 更新参照。

**锁定参考图**: `design/ux/mockups/van-interior/van-interior-target-dark-2026-06-13.png`（PM 2026-06-13 选定，AI 生成的概念目标）。

- **整体**: 拥挤、**无窗**的政府报废接驳车内舱；**暗青灰做旧钣金**（市政青 × 死黑橡胶），锈渍渗缝、油污刮痕、铆钉外露、钢板地面。**冷**单管荧光，强剪影、lo-fi、轻颗粒——黑暗中可读（致命公司式「做法」参考，**不抄其飞船/资产/UI**）。PM 明确否决「干净银白/科幻舱」方向。
- **座位**: 两排长椅沿车壁、隔中央过道**面对面**坐（对应 `VanCabin` 4 座）；玩家**第一人称**坐姿，转头即见对排与身边队友——这是「看见彼此 + 对话」的社交舞台（近距语音 `ProximityVoiceChat` 已在）。
- **道具（civic-paperwork 身份）**: 车尾门贴催缴/安全告示 + 红章；隔板派工夹板；头顶扶手 + 立杆；调度电台带小绿指示灯；地面油污。返程时生态柱货物可同舱（参考图地面那根管状物即此意象）。
- **基调护栏**（creative-director）: 羁押车厢、不是休息室——冷光、机构脏旧，**不加**任何「大厅」装饰（表情/点唱机/小游戏/炫装）。社交感**涌现**自「面对面 + 近距语音 + 一起等」，不靠加社交功能。

**否决项**: 旧 `11_transit_strip` 那版「2D 票据条盖在空黑盒上」的呈现已被 PM 否决——透视/构图/2D 取舍以本参考图 + UX 评审结论为准（顶部条最小化，信息尽量走实体 diegetic）。

> ⚠ **art-bible 待修（banach/art-director）**: §State 2 现写有「车窗外景」，与锁定的**无窗**车厢矛盾（无窗对「加载伪装」与压迫感都更好）。应改「无窗」为锁定意图，并把暗青灰内饰色板写入 art-bible 车厢段。

## States & Variants

| State / Variant | Trigger | What Changes |
|---|---|---|
| 等人（Boarding） | E 上车坐定 | 票据条=就位 N/M；X 可离座 |
| 全员就位待签 | `AreAllLivingSeated` 为真 | 中央弹派车单「待签发」；房主见 Space 提示，非房主见「等待房主签发」 |
| 已签发→在途 | 房主 Space | 章砸落「已签发」+ StampThunk → 卡 0.6s 后收起；票据条切里程态；X 失效 |
| 在途·加载超时 | 里程到 92% 但加载未完 | 条停在尾段呼吸（±2%），文案「即将抵达」；永不显示"加载" |
| 收工申请填写中 | 未取柱+房主 Space | 房主=申请单卡；其余 peer=票据条附加行「房主正在填写…」 |
| 申请撤回 | Esc（墨条未满） | 卡收起回等人态；附加行消失 |
| 返程在途 | 任一终态发车 | 同在途；~1.5s 后结算单另卡叠出（结算 spec，深度 -100 在票据条之上） |
| 到站开门 | 加载完成 且 最短在途走完 | 引擎熄火→0.5s 静默+squelch→门开缝渗光（HQ 钨丝暖/塔楼阴天灰）→门全开 E/走出；票据条「已抵达」2s 后收起 |
| 失败强制返程 | all-downed | 无上车/派车单；直接车厢+在途+结算单（演出从简） |
| 离线单人 | 无网络 Play | 就位 1/1 即「全员就位」；派车单照常（仪式保留）；收工申请仍走签字（防误触同样适用） |
| 任务定义缺失 | task=null fallback | 委托名退化场景名（同结算 spec），目的地行隐藏 |

无 loading 态——本流程**就是** loading 态的化身。

## Interaction Map

输入域：键鼠（PC 单平台，手柄 TBD）。

| 动作 | 输入 | 即时反馈 | 结果 |
|---|---|---|---|
| 上车 | 准星对车尾 `E` | 落座+车厢显示+票据条 | RequestSeat（同步） |
| 离座 | `X`（仅 Boarding） | 票据条收起（本地离座） | RequestLeaveSeat |
| 签发发车 | `Space`（房主，需全员就位/返程不限） | 「已签发」章砸落+StampThunk | 出发=BeginMission；完整返程=RequestDepart |
| 发起收工申请 | `Space`（房主，未取柱） | 申请单滑入（纸声） | 纯本地卡显示，无状态变更 |
| 签字确认 | 按住 `E` 1.2s | 签字栏墨条填充+完成时落章音 | RequestDepart（PartialReturn 路径） |
| 撤回申请 | `Esc` | 卡下滑收起 | 回等人态，可重复发起 |
| 下车 | 到站门开后 `E` 或直接走出 | 舱体拆除 | 回到场景控制 |

冲突规避：派车单/申请单显示期间玩家已坐定（移动输入本就锁定），E 键无相邻可交互物；
结算单的 E=收起与申请单的按住 E 不会同屏（申请单只在终态前出现）。

## Transitions & Animations

- **派车单进场**: 全员就位触发，自下滑入 0.25s（纸声，同结算单语法）。
- **签发**: Space → 「已签发」章砸落（缩放 1.3→1.0，0.12s）与 StampThunk 同帧 → 0.6s 后卡下滑收起，票据条接力。
- **申请单签字**: 按住 E 期间签字栏青墨匀速填充 1.2s；中途松手 0.3s 内倒流清空；填满→落章音+卡收起+发车。
- **在途墨条**: 匀速填充至 92%；加载未完则 ±2% 呼吸；完成补满到 100%（0.4s）。
- **到站开门时序（PM 锁定）**: 引擎声收束熄火 → 0.5s 静默+电台 squelch → 车尾门开缝、
  新场景光渗入（HQ=钨丝暖 #FFE / 塔楼=阴天钢灰 #B8C4CE——光色即地点签名）→ 门全开。
  全链无黑帧无硬切。
- **减弱动效**（设置 spec 依赖）: 卡即显/即隐、章不缩放、墨条无呼吸（直接显示「即将抵达」文字）、开门渗光保留（空间事件非动效装饰）。

## Accessibility

- 「待签发/已签发」以章内文字传达，非仅颜色；警告行以 ⚠ 符号+文字传达，非仅红色。
- **按住签字的运动负担**: 1.2s 按住对部分玩家是障碍——设置 spec 需收「按住改双击切换」
  无障碍开关（与 HUD spec 按住进度条同一开关，归口设置 spec）。
- 申请单无时限：填写期间车不会自己开走；Esc 随时撤回，无键盘陷阱。
- 全键盘可达（E/X/Space/Esc），无指针依赖。
- 墨字/纸底对比度沿用纸卡标准 ≥4.5:1（主菜单 spec 锁定值）；票据条在途 % 数字 ≥14px 等效（HUD spec 下限）。
- 到站静默 0.5s 期间保留电台 squelch 声学提示（纯视觉变化前先给听觉锚点）。

## Localization Considerations

- 最长行=申请单警告行（约 20 全角字符）与票据条头行（委托名·地点拼接，委托名超 12 字截断加 …）。
- 英文扩展 ~40%：派车单标签列固定宽、值列右对齐换行（HIGH PRIORITY：「提前收工折算」行同结算单处理）。
- 条款号 B-2 不本地化（与结算 spec 同一规则，全语言一致）。
- 金额 `±NG`、百分比 `N%` 千位不分隔。
- 「已签发/待签发/已抵达」章文为 2-3 全角字符设计；英文章文（DISPATCHED 等）需缩字号适配章框。

## Acceptance Criteria

- [ ] 出发全链（接单→上车→签发→在途→开门→任务点站定）全程无黑屏、无 2D 加载页、无 >0.5s 帧冻结
- [ ] 全员就位前房主 Space 无效且派车单不出现；最后一名队友就位后 1s 内所有 peer 弹出派车单
- [ ] 未取柱时房主 Space 弹申请单而非直接发车；按住 E 不足 1.2s 松手不生效；Esc 撤回后可再次发起
- [ ] 申请单折算预估金额与最终结算单实付一致（同一计算器路径）；完整度 <50% 时拒收警告行出现
- [ ] 队友在房主填写申请期间 1s 内看到票据条附加行；撤回后附加行消失
- [ ] 在途进度条匀速无回退；加载超过最短在途时间时条停 92% 呼吸且不出现"加载"字样
- [ ] 到站时序按锁定顺序播出（熄火→静默→渗光→全开）；门全开前玩家无法离舱
- [ ] X 离座仅发车前有效；签发后在途中无任何离舱路径
- [ ] 减弱动效开启时：无章砸落缩放、无墨条呼吸，流程功能完整
- [ ] 离线单人：1/1 即全员就位，派车单与申请单流程完整可用

## Open Questions

1. **异步加载重排实施** — 签发即 LoadScene、SceneEvent 完成+最短在途双门控开门：出发与返程
   两处管线重构，是本 spec 实施量最大项；NGO 场景事件在客户端的进度可见性需验证。
2. **失败强制返程演出** — 全员倒地直接入舱是否需要最简"事故回收"过场（黑屏例外？拖拽演出？）
   ——倒地视角到车厢的衔接现状是硬切，与"无黑屏"锁定的张力待 PM 裁决。
3. **最短在途秒数** — `dispatchTransitSeconds` 现值沿用还是按"读结算单所需时间"重调
   （返程建议 ≥10s，给结算单阅读窗口）；出发/返程是否分开两个 knob。
4. **申请填写中同步** — NV bool vs ClientRpc 通知，实施时由 laplace/网络层定。
5. **按住改切换无障碍开关** — 归口设置 spec（与 HUD 按住进度条共用一个开关）。
6. **派车单单号** — 与结算单单号同一派生规则（账本 ADR 一并定）还是不带单号。
7. **新模式入库** — 「派遣票据条」加入 interaction-patterns.md 种子清单（累计 8 个：
   终端 3 + HUD 3 + 条款式账目行 + 本条）。
8. **player-journey.md 缺失** — 与前三屏同一缺口。
9. **手柄映射 TBD** — Space/X/按住 E 的手柄等价键位，归口全局输入表。
