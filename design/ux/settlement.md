# UX Spec: 委托结算单 (Mission Settlement Card)

> **Status**: Approved（PM 终审通过 2026-06-11）
> **Author**: Yan Dai + ux-designer
> **Last Updated**: 2026-06-11
> **Journey Phase(s)**: 任务收尾 → 返程 → HQ 入账（无 player-journey.md，按核心循环推定）
> **Template**: UX Spec
> **统一语言定位**: 三类表面之「盖章公文卡」（结算=委托结算单）
> **队列**: UI 重设计第 3 屏（电脑✅ → HUD✅ → **结算** → 上车+加载 → 大厅 → 图谱 → 设置）

---

## Purpose & Player Need

玩家拉杆发车后想知道一件事：**"我们到底拿到多少钱，被扣了多少，凭什么扣。"**
结算单在返程车厢内递到每人面前，给出结论章（完成/部分结算/失败）、逐项金额拆解、
以及客户使用备注——本作讽刺的法定载体（pillar 锁定：讽刺走结算单条款与备注，
不走旁白）。没有这张单子，结算就只是 HQ 电脑上一个数字跳动，"被市政条款克扣"
的核心体验完全丢失。

**PM 锁定（2026-06-11）**: 出现时机=返程车厢内（车厢即加载——被俘获的在途时间正好
读单子）；明细=逐项拆解（income → deductions → net，扣项条款化）。
**经济 GDD 锁定**: 玩家只看钱——声望/经验为内部数值，**单上不得出现**。

---

## Player Context on Arrival

- **时点**: 拉杆发车或部分返程确认后 ~1.5s，运输舱启动、引擎声起、电台底噪中。
- **刚经历**: 完整任务循环的收尾——扛柱狂奔/减员撤退/空手而归。肾上腺素回落段。
- **情绪假设**: 松弛+余悸，协作复盘欲强（"刚才那下磕的"）。结算单是话题的物证。
- **到达方式**: 非自愿——单子是"递"到面前的（模态出现），不是玩家查询的。
- **多人**: 每个 peer 各自收到同一张单（数据同步自 host），各自阅读、各自收起。

---

## Navigation Position

不是导航目的地，是事件驱动的模态卡：
`任务场景 → 拉杆/确认返程 → 返程车厢（VanTransitOverlay）→ 【委托结算单】→ 到站 → HQ`
唯一入口是任务终态；无任何菜单可主动打开它。**到站后的回看走办公电脑「公司账本」
页签**（终端 spec 已锁定账本回看结算讽刺文本）。

## Entry & Exit Points

| Entry Source | Trigger | Player carries this context |
|---|---|---|
| 拉杆发车（Delivered） | `TowerVanDepartLever` → `TowerMissionManager.RequestDepart` 通过 | 终态 Delivered，完整度，货物在舱 |
| 部分返程确认（PartialReturn） | `MissionVanExitPoint` 确认流程 | 终态 PartialReturn，完整度警告已读过 |
| 全员倒地（Failed） | all-downed poll → Failed → 返程 | 终态 Failed |

| Exit Destination | Trigger | Notes |
|---|---|---|
| 收起（卡折叠，车厢继续） | 按 `E` 或 `Esc` | 仅本地；`MvpPendingReward` 维持 pending，不入账 |
| 重看（车厢内） | 收起后按 `Tab` | 仅返程在途期间有效 |
| 到站下车 | 场景切换完成、舱门开 | 卡若未收起则随下车自动收起；**入账动作不在本卡**——回所后办公电脑按 E 落章入账（终端 spec 待领赏金块） |

不可逆点：无。本卡纯只读；真正的状态变更（入账）发生在办公电脑。

## Layout Specification

### Information Hierarchy

1. **结论章** — 完成 / 部分结算 / 失败（印章红斜章，3 秒判读的第一眼）
2. **实付金额** — 单上最大的数字
3. **扣项明细** — 逐行条款化扣减（讽刺主载体之一）
4. **客户使用备注** — 讽刺主载体之二（手写体短评）
5. **领款指引** — 页脚一行：回所后凭单至办公终端入账

### Layout Zones

竖版 A5 纸感卡片，屏幕中央偏下（不挡车窗外加载景）：

- **页眉**: 市政青横带——公司抬头「黑色委托事务所」+ 单号 + 委托名
- **明细表**: 纸底墨字账目行（收入行 / 扣项行带条款号 / 合计实付行）
- **结论章**: 印章红斜章叠在明细表右上角（微旋转 −8°，叠墨不遮数字）
- **备注块**: 虚线框「客户使用备注」手写体
- **页脚**: 领款指引 + 收起键提示

### Component Inventory

| 组件 | 类型 | 内容 | 交互 | 模式 |
|---|---|---|---|---|
| 公文卡容器 | 卡片 | 纸 #D6CCAE 底、市政青 #3F5F5C 页眉框线 | 否 | **盖章公文卡**（主菜单 spec 已锁全游戏模态语法） |
| 账目行 | 文本行 ×N | `项目名 …… ±NG`，点线引导 | 否 | 新模式：**条款式账目行** |
| 结论章 | 印章块 | 完成/部分结算/失败，印章红 #C23A2B | 否 | 盖章公文卡的章位 |
| 客户备注 | 文本块 | 1–3 行手写体讽刺短评（按任务+结果） | 否 | — |
| 页脚指引 | 文本行 | 「回所后凭本单至办公终端入账」+ `[E] 收起` | E/Esc 收起 | — |

### 各结果变体的账目行

| 结果 | 行 | 来源 |
|---|---|---|
| **完成** | `委托报酬 +300G` → `运输磕碰扣损(条款C-7) −NG`（注完整度%与每次−3%）→ `实付 NG` | 基础报酬 × `SyncedCompleteness`，扣损=差额 |
| **部分结算** | `提前收工折算(条款B-2) +66G`（注：按报酬 22% 折算，不低于慰问金）→ `实付 66G` | `PartialMoneyFraction` 0.22 / max(慰问金) |
| **失败** | `慰问金(条款D-1) +20G` → `实付 20G` | `failureConsolationMoney` |

完整度无扣损（100%）时扣损行省略，只列报酬+实付。
（加班/错件扣款行的语法保留给未来计时类委托——塔楼任务无时钟，本版不出现。）

## States & Variants

| State / Variant | Trigger | What Changes |
|---|---|---|
| 完成 | Delivered | 章=完成；含扣损行（如有） |
| 部分结算 | PartialReturn | 章=部分结算；折算行 |
| 失败 | Failed | 章=失败；仅慰问金行；备注换失败语气 |
| 离线走查（PreviewWalker） | 非联机 Play | 同样显示（管理器已有离线路径），数据本地 |
| 任务定义缺失 | task=null fallback | 委托名退化为场景名；金额走 `MissionRewardFallbacks`；备注块隐藏 |
| 备注文本缺失 | 任务无备注配置 | 备注块整体隐藏，不留空框 |

无 loading/empty 态（数据在卡出现前已是终值）。多人各端数据一致（同步 NV + 同一计算器）。

## Interaction Map

输入域：键鼠（PC 单平台，手柄 TBD）。本卡只读，交互极简：

| 动作 | 输入 | 即时反馈 | 结果 |
|---|---|---|---|
| 收起 | `E` 或 `Esc` | 纸滑出音 + 卡下滑 | 卡折叠，车厢视野恢复；仅本地 |
| 重看（在途） | `Tab` | 纸滑入音 + 卡上滑 | 卡重现，数据不变 |

卡显示期间不锁移动输入（车厢内本就坐定）；不拦截语音/聊天。

## Transitions & Animations

- **进场**: 卡自下滑入 0.25s（纸声）；0.4s 后结论章砸落（缩放 1.3→1.0，0.12s）**与
  `SynthAudio.StampThunk` 同帧**（现实装为 `ApplyResultLocally` 即时播放——实施时把音效
  触发移到章落帧，或卡出现对齐音效时刻）。
- **退场**: 下滑 0.2s。
- **减弱动效**（设置 spec 依赖）: 开启时进退场即显/即隐，章不做砸落缩放。

## Accessibility

- 结论不止靠章的颜色：章内文字（完成/部分结算/失败）即结论本身；失败态备注语气变化为辅。
- 墨字/纸底对比度待实测（与 HUD spec 同一开放问题，目标 WCAG-AA）。
- 卡**无阅读时限**——到站不强制弹掉之前玩家可一直读；收起后在途可 Tab 重看，回所后账本回看。
- 全键盘可达（E/Esc/Tab），无指针依赖，无键盘陷阱。
- 章砸落动画有减弱动效替代（见 Transitions）。

---

## Localization Considerations

- 中文为主文案。最长行=扣项行（项目名+条款号+金额，约 22 全角字符）——卡宽按此设计；
  英文扩展 ~40% 时条款名换行至缩进第二行，金额列右对齐不动（HIGH PRIORITY）。
- 金额统一 `±NG` 格式，千位不分隔（数额量级 ≤4 位）。
- 条款号（C-7/B-2/D-1）为虚构稳定标识，**不本地化**——全语言一致，方便社区梗化。
- 手写体备注：中文需手写风字体支持；英文 fallback 用斜体。

## Acceptance Criteria

- [ ] 发车/确认返程/全员倒地三种终态后 2s 内，车厢内每个 peer 都收到结算单，结论章与 host 判定一致
- [ ] 完成态且完整度 <100% 时，扣损行金额 = 基础报酬 − 实付，与 `MissionRewardCalculator` 输出一致（含 0 扣损时省略该行）
- [ ] 单上不出现声望、经验或任何内部数值；实付与回所后办公终端待领赏金块金额一致
- [ ] `E`/`Esc` 收起、`Tab` 重看在途均生效；到站自动收起后电脑账本可回看同一单
- [ ] 任务定义缺失（fallback 路径）不报错：委托名退化、备注块隐藏、金额走 fallback 表
- [ ] 结论以章内文字传达（非仅颜色）；减弱动效开启时无缩放砸落动画
- [ ] 卡出现到可读 ≤0.5s（无资产加载等待——纯程序化/预载字体）

## Open Questions

1. **`MissionRewardResult` 行项扩展** — income/deduction 拆解数据现不存在，需小幅扩展结构
   （计算器内补 BaseMoney/Deduction 字段 + 测试），实施批次处理。
2. **客户使用备注字段与文案** — `OfficeTaskDefinition` 加按结果分支的备注组；文案需 writer
   pass（讽刺基调：市政腔+客户傲慢，每任务 ×3 结果 ×1–3 条）。
3. **单号生成规则** — 建议 `BC-2098-NNNN`（存档累计单数派生），与账本 ADR 一并定。
4. **逐单结算历史 ADR** — 与办公电脑终端 spec 共用（账本页签回看）；本卡入账时写入归档。
5. **StampThunk 音效时刻对齐** — 现为结算应用即播；改为章落帧触发（纯实施细节）。
6. **分析事件** — 项目无 telemetry；若立项，`settlement_shown` 建议字段见 Events 节。
7. **新模式入库** — 「条款式账目行」加入 interaction-patterns.md 种子清单（现累计 7 个）。
8. **player-journey.md 缺失** — 与 HUD spec 同一缺口。
